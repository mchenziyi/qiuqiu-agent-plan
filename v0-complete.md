## 🎯 V0 目标回顾

用 4 周时间，跑通 Agent 最核心的循环：**LLM → 调用工具 → 拿到结果 → 再问 LLM → 直到完成**。

## 🧱 V0 做了什么

### 核心：一个能跑起来的 Agent

```go
type Agent struct {
    tools    map[string]Tool           // 注册的工具
    messages []ChatCompletionMessage   // 对话历史
}

func (a *Agent) Run(ctx, 用户问题) (回答, 错误) {
    记录用户问题到历史

    for 最多 10 轮 {
        把"历史 + 工具列表"发给 LLM

        if LLM 没要求调工具 {
            return LLM 的回答  // 任务完成
        }

        if LLM 要求调工具 {
            执行工具 → 把结果放回历史  // 继续循环
        }
    }

    return 超时错误
}
```

**整个 Agent 就这么一个循环。没有更复杂的了。**

### 四个内置工具

| 工具 | 一句话功能 | 为什么需要它 |
|------|-----------|------------|
| `read_file` | 读文件内容 | LLM 不能直接访问文件系统 |
| `write_file` | 写内容到文件 | LLM 不能直接保存东西 |
| `list_directory` | 列出目录内容 | LLM 不知道项目里有什么文件 |
| `run_shell` | 执行命令 | 兜底：上面三个搞不定时用 |

### 交互方式

```
你: 创建一个 hello.txt 写你好
球球: 调用 write_file → 完成 ✅

你: 刚才写的什么？
球球: 调用 read_file → 回答"你好"（它记得历史）
```

## 🧠 V0 学到的核心知识

### ① Agent 不是什么高深的东西

就是：

```text
注册几个工具 → 循环 { 问 LLM → LLM要调工具就调 → 再问 LLM → ... → LLM说完成了就结束 }
```

**没有神秘的概念，就是一个 for 循环。**

### ② LLM 不会"主动"做事

LLM 只是**建议**："我想调 write_file，参数是……"
真正执行工具的是**你的代码**。控制权在你手上。

### ③ 连续对话是怎么实现的

每次把 `messages` 累积起来，下次全量发给 LLM。LLM 看到历史，就能"记住"之前说过什么。

```text
第 1 轮：messages = [用户说"创建...", LLM的回答, 工具结果, LLM的回答]
第 2 轮：messages = [第1轮的完整内容] + [用户说"刚才写的什么"]
         ↓ LLM 看到全部历史，知道"刚才"指的是什么
```

**不是 LLM 有记忆，是你的代码帮它记住了。**

### ④ 工具设计就两个要点

| 要点 | 说明 |
|------|------|
| **名字要清楚** | `list_directory` 比 `run_shell("dir")` 好——LLM 不用猜"列目录该用什么命令" |
| **返回值要清楚** | 返回"文件 xxx 的内容是……"比返回原始字节好——LLM 直接拿去用 |

把 LLM 当成人来设计工具：**给它见名知意的名字，给它一看就懂的返回值。**

### ⑤ 专用工具 vs 通用工具

| 类型 | 例子 | 优点 | 缺点 |
|------|------|------|------|
| 通用工具 | `run_shell` | 什么都能干 | LLM 要猜参数，容易出错 |
| 专用工具 | `read_file` | LLM 不用猜，安全，可控 | 只能干一件事 |

平衡之道：**优先用专用工具**兜不住的事再用通用工具。

## 🔜 下一步：V1 — Planning

V0 的 Agent 能做简单任务了（读文件、写文件、列目录），但它有个大问题：

> **复杂任务一步做不完。**

比如"给我的 Go 项目加一个健康检查接口"——它不知道要先分析路由、再写代码、再测试。它只会"一次性"去做，然后做一半就乱套了。

V1 就是解决这个问题的：**让 Agent 学会拆任务。**
